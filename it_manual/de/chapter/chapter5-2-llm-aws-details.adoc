===== Schritt-für-Schritt: AWS Bedrock aufsetzen

*Schritt 1: AWS Bedrock aktivieren*

* AWS Console öffnen: https://console.aws.amazon.com
* Zu Bedrock navigieren: in der Suche „Bedrock“ eingeben
* Region wählen: z. B. eu-central-1 (Frankfurt) oder us-east-1
* Model Access aktivieren:
** Links auf Model access klicken
** Manage model access
** Die benötigten Modelle auswählen (z. B. OpenAI OSS, Meta, Anthropic etc.)
** Save changes klicken

*Hinweise:* 

* Bei manchen 3rd‑Party-Anbietern (z. B. Anthropic) dauert die Freischaltung 5–15 Minuten
* Model Access ist zwingend notwendig. IAM allein reicht nicht – Modelle müssen manuell aktiviert werden.
* Model‑IDs müssen exakt stimmen. Fehlerhafte oder veraltete IDs erzeugen AccessDeniedException.
* Region muss konsistent sein. Modellregion, Bedrock API‑Region und IAM‑Policy müssen übereinstimmen.
 
*Schritt 2: IAM-Berechtigungen einrichten*

Damit eine EC2‑Instanz (oder andere AWS‑Services) auf Bedrock zugreifen können, werden IAM-Role *oder* IAM-User verwendet.

* IAM-Role (empfohlen für AWS‑Ressourcen)
* IAM-User mit Access Keys (empfohlen für lokale Entwicklerumgebungen)

|===
|Methode|Vorteile|Nachteile

|IAM-Role
| Automatische, sichere temporäre Credentials via IMDS (Instance Metadata Service), Keine Schlüsselverwaltung
| Nur für AWS‑Ressourcen geeignet (nicht für Entwickler lokal)

|IAM-User
| Für lokale Tools & Entwickler nutzbar, auch außerhalb von AWS
| Dauerhafte Schlüssel müssen sicher gespeichert & rotiert werden
|===

*Schritt 3: Erstellen IAM‑Role oder IAM‑User*

Für den Zugriff auf AWS Bedrock muss entweder eine IAM‑Role oder ein IAM‑User erstellt werden. +
Benutzerdefinierte IAM‑Policies müssen *vor* der Erstellung der IAM‑Role oder des IAM‑Users erzeugt werden, damit diese während der Erstellung ausgewählt und zugewiesen werden können.

Die minimale Policy für Bedrock‑Zugriff ist:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "bedrock:InvokeModel",
        "bedrock:InvokeModelWithResponseStream"
      ],
      "Resource": "arn:aws:bedrock:*::foundation-model/*"
    }
  ]
}
```
→ Optional: Regionale Einschränkung, z. B. nur Frankfurt:
```
"Resource": "arn:aws:bedrock:eu-central-1::foundation-model/*"
```

*Amazon ARN* (Amazon Resource Names) sind eindeutige Bezeichner, mit denen AWS‑Ressourcen wie KI-Modelle eindeutig identifiziert werden. +
Siehe: `https://docs.aws.amazon.com/IAM/latest/UserGuide/reference-arns.html`

*ARN-Beispiel eines Foundation Models:*
```
arn:aws:bedrock:eu-central-1::foundation-model/openai.gpt-oss-20b-1:0
```
*Bestandteile:*

* arn: – Kennzeichnung des Ressourcenformats
* aws: – AWS Namespace
* bedrock: – Dienst
* eu-central-1: – Region
* foundation-model: – Ressourcentyp
* openai.gpt-oss-20b-1:0 – Modell-ID

Neuere Modelle nutzen *Inference‑Profile*, um regionsübergreifende Lastverteilung zu ermöglichen. +
siehe https://docs.aws.amazon.com/de_de/bedrock/latest/userguide/inference-profiles-support.html

Der Aufruf eines Inferenceprofiles enthält immer die Quellregion (`eu-central-1`), den Profileaufruf (`inference-profile`) und die Zielregionen (`eu`) und das Modell (`anthropic.claude-sonnet-4-6`).

*ARN-Beispiel eines Inference‑Profils:*
```
arn:aws:bedrock:eu-central-1:<account-id>:inference-profile/eu.anthropic.claude-sonnet-4-6
```
*Wesentliche Bestandteile:*

* Quellregion - eu-central-1
* Konto-ID - <account-id>
* Ressourcentyp - inference-profile
* Profil/Modell - eu.anthropic.claude-sonnet-4-6 +
(verteilt auf mehrere EU‑Regionen: eu-central-1, eu-north-1, eu-south-1, eu-south-2, eu-west-1, eu-west-3)

Policy für EU‑weite Nutzung von foundation‑models + inference‑profiles
```
{
      "Version": "2012-10-17",
      "Statement": [
            {
                  "Effect": "Allow",
                  "Action": [
                        "bedrock:InvokeModel",
                        "bedrock:InvokeModelWithResponseStream"
                  ],
                  "Resource": [
                        "arn:aws:bedrock:eu-*::foundation-model/*",
                        "arn:aws:bedrock:eu-*:<account-id>:inference-profile/*"
                  ]
            }
      ]
}
```
*Wichtig:*

* eu-* für foundation-model, da Inference‑Profile automatisch EU‑weit verteilen
* eu-*:<account-id> für Inference‑Profile, da diese konto-gebunden sind

*Schritt 4: Access Keys generieren (entfällt bei IAM-Role)*

1. AWS IAM Console → Users → Ihr User → Security credentials
2. Create access key
3. *Notieren Sie für die spätere Konfiguration in der Applikation:*
   * AWS Access Key ID
   * AWS Secret Access Key
   * AWS Region
   * Model ID (z. B. `openai.gpt-oss-20b-1:0`)

*Beispiel:*
```bash
AWS Access Key ID: AKIAIOSFODNN7EXAMPLE
AWS Secret Access Key: wJalrXUtnFEMI/K7MDENG...
AWS Region: eu-central-1
Model ID: openai.gpt-oss-20b-1:0
```

*Zuteilen der IAM-User / IAM-Role zur EC2-Instanz*

*IAM Role*: +
Rolle der EC2‑Instanz zuweisen über → EC2 → Instance → Actions → Security → Modify IAM role. +
*IAM User*: +
Für einen IAM‑User sind die AWS‑Credentials für die EC2 Instanz in der Konfiguration der Applikation hinterlegt.
